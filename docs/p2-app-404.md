# P2 — the app renders a 404 and Vercel stores it

**Whose bug:** application code (+ a Nitro docs gap). **Symptom:** a 404
served with `x-vercel-cache: HIT` whose body names the **real** path (not
`-isr`). **No Nitro or Vercel bug involved.**

Prerequisite reading: [how ISR works on Vercel](./how-it-works.md), in
particular the [measured caching facts](./how-it-works.md#the-load-bearing-caching-facts-measured).

## The mechanism

The ubiquitous app pattern:

```ts
if (!story.value) throw createError({ statusCode: 404, fatal: true })
```

turns a *transient* failure (CMS rate-limit, timeout) into a 404 — which
Vercel stores as valid ISR output (rule 3). Under `isr: true` the stored 404
**never expires**, so one bad render breaks the route until a purge or
redeploy.

## Reproduce it

```sh
./scripts/poison-demo.sh https://<deployment>.vercel.app
```

`/flaky` (`isr: true`) and `/flaky-ttl` (`isr: 60`) render the same page,
whose SSR fails 50% of the time.
`POST /api/invalidate {"tags":["page:/flaky"]}` simulates CMS-publish purges,
forcing re-populations. Observed live 2026-07-18:

- one bad re-population stored a 404 into the never-expiring `/flaky` entry →
  **`404` + `x-vercel-cache: HIT` on every request, permanently**;
- `/flaky-ttl` — the identical page with a finite TTL — healed on a
  revalidation retry within ~2 minutes.

Standalone minimal version:
branch [`demo/p2-app-404`](../../tree/demo/p2-app-404).

## The trap: `useAsyncData` collapses "error" and "empty"

P2 is easy to write because `useAsyncData`/`useFetch` (and CMS-client wrappers
built on them) collapse *"upstream threw transiently"* and *"resource
genuinely absent"* into the same nullish `data` — so
`if (!data.value) throw createError({ statusCode: 404 })` misclassifies a
429/timeout as a permanent 404. Distinguish them by inspecting `error.value`
first:

```ts
const { data, error } = await useAsyncData(key, fetcher)
if (error.value) {
  if (error.value.statusCode === 404) throw createError({ statusCode: 404, fatal: true })
  throw createError({ statusCode: 503, fatal: true })  // transient → never cached, stale kept
}
if (!data.value) throw createError({ statusCode: 404, fatal: true })  // genuinely absent
```

(With many CMS clients a missing resource arrives as a *thrown* 404 on
`error.value`, not as empty `data` — inspect the error, don't rely on
`!data.value`.) A `throw createError({ statusCode: 503 })` during SSR renders
as a real 503, which Vercel does not cache and which leaves any existing good
copy in place.

## Fix

Never combine `isr: true` with fallible rendering, and **throw 5xx, not 404,
for transient failures** — the "harsher" status is the safe one. Full
confidence-graded guidance: [fixes → P2](./fixes.md#p2--app-rendered-404).
